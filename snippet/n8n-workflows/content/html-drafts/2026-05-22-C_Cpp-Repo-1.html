The Breakdown

  • What it does: Acquires a load mutex, applies recipe-specific NPU eviction rules and server caps, releases the lock to load the model outside the critical section, then re-acquires to confirm success or evict all servers and retry with a fallback.
  • Key responsibility: Manages the NPU/GPU server lifecycle in Lemonade's model router, ensuring hardware resources aren't double-allocated across incompatible recipes (ryzenai-llm, whispercpp, flm) while allowing concurrent inference on already-loaded models to proceed unblocked.
  • The clever part: The mutex is released before new_server->load() because model loading takes seconds — holding the lock would block all concurrent requests. The re-acquire after load ensures safe writes, while the retry path evicts all servers first to guarantee a clean state before the second attempt.
